Обычная `map`-а Go **небезопасна для параллельного использования** при одновременном чтении и записи.

Для обеспечения потокобезопасности обычные `map`-ы часто используются в сочетании с **`sync.Mutex`** или **`sync.RWMutex`**.

**`sync.Map`** — это **не универсальная замена** для всех сценариев параллельной работы с `map`-ами. В большинстве случаев предпочтительнее использовать обычную `map`-у Go с блокировкой.

### Когда использовать `sync.Map`

Тип `sync.Map` оптимизирован для двух распространенных случаев использования:

1. Когда запись для заданного ключа записывается только один раз, но читается многократно. Типичный пример — **кеш**, где записи добавляются, но никогда не удаляются и не изменяются.
2. Разные горутины работают с **разными, непересекающимися наборами ключей**. Например, Горутина A работает только с ключами, начинающимися на "A", а Горутина B — с ключами, начинающимися на "B".

В этих двух случаях использование `sync.Map` позволяет **значительно снизить конфликты блокировок** (_lock contention_) по сравнению с обычным `map` + `sync.Mutex` или `sync.RWMutex`, что приводит к лучшей общей производительности в условиях высокой конкуренции.

### Когда стоит избегать `sync.Map`:

- Если в сервисе **больше операций записи, чем чтения**.
- Если требуются **более сложные операции**, которые `sync.Map` плохо поддерживает.
- В таких случаях `sync.Map` может привести к **снижению производительности** (как по памяти, так и по CPU), поскольку внутренне она использует две `map`-ы.

### Недостаток `sync.Map`:

- **Потеря типобезопасности**: `sync.Map` хранит ключи и значения как **`interface{}`**, в отличие от [[Golang. Тип Map в Go|обычных мап]], где выше типобезопасность.

## Что такое `sync.Map`

Когда несколько горутин записывают или читают, `sync.Map` берет на себя все блокирующие (или атомарные) операции, без необходимости ручной блокировки и гонки данных. Вдобавок, операции чтения, записи и удаления работают быстрее, чем в [[Golang. Тип Map в Go|обычной]] `map` с [[Golang. Примитивы синхронизации#Mutex|мьютексом]].

Запись в `sync.Map` всегда **синхронизируется** перед последующим чтением. Это гарантирует, что читатели не увидят старое значение.

### Основные методы и их атомарность.

Все методы `sync.Map` являются **атомарными** (т.е. потокобезопасными).

#### Основные операции

- **`Load(key)`**: Извлечение значения по ключу.
- **`Store(key, value)`**: Сохранение пары ключ-значение.
- **`Delete(key)`**: Удаление пары по ключу.
- **`Clear()`** (Go 1.23+): Удаляет все элементы из мапы.
- **`Swap(key, value)`** (Go 1.20+): Заменяет старое значение и возвращает его.

#### Условные операции

- **`LoadOrStore(key, value)`**: Возвращает существующее значение, если ключ есть; иначе сохраняет новое. Удобно для операций "get or create".
- **`LoadAndDelete(key)`**: Извлекает значение по ключу, а затем удаляет его.
- **`CompareAndDelete(key, old)`** (Go 1.20+): Удаляет ключ, только если его текущее значение совпадает с `old`.
- **`CompareAndSwap(key, old, new)`** (Go 1.20+): Обновляет значение, только если текущее значение совпадает с `old`.

#### Итерация (`Range`)

- **`Range(f func(key, value any) bool)`**: Метод для итерации по мапе.
- Он решает проблему **фатальной ошибки при итерации** (`fatal error: concurrent map iteration and map write`) по обычной мапе, в которую одновременно идет запись.
- **`Range` не блокирует мапу** на все время итерации. Это позволяет другим горутинам **одновременно** добавлять, обновлять или удалять записи.
- **Компромисс**: Из-за отсутствия полной блокировки, итерация **может не предоставить абсолютно согласованный ("идеальный") снимок** мапы на момент начала обхода.
- Итерация останавливается, если функция `f` возвращает `false`.

## Как работает `sync.Map`

```go title:sync/map.go ln:40
type Map struct {
	_ noCopy // Marker to forbid copying the Map value after first use (enforced by go vet)

	mu Mutex // Lock that protects the 'dirty' map and the 'misses' counter

	read atomic.Pointer[readOnly] // Atomic pointer to the read-only map, allowing lock-free reads
	dirty map[any]*entry // Map containing recent writes/updates, guarded by 'mu'
	misses int // Counter for when a Load operation fails to find a key in 'read' (guarded by 'mu')
}
```

### Внутреннее устройство: `readonly` и `dirty`

`sync.Map` использует две обычные `map` (`map[any]*entry`) для разделения операций чтения и записи:

1. `readonly` Map:
   - **Назначение:** Быстрые операции чтения (`Load`).
   - **Доступ:** Доступ осуществляется через **`atomic.Pointer`**, что позволяет **lock-free** (без блокировок) чтение для множества горутин.
   - **Особенность:** Содержит **стабильный** набор данных, но может быть **неактуальной** (отсутствуют самые новые записи).
   - **Флаг `amended`:** Поле в структуре `readOnly`, которое равно `true`, если в мапе `dirty` есть хотя бы один ключ, отсутствующий в `readonly`.

```go title:sync/mutex ln:77
// readOnly is an immutable struct stored atomically in the Map.read field.type
readOnly struct {
    m       map[any]*entry
    amended bool // true if the dirty map contains some key not in m.
}
```

2. `dirty` Map:
   - **Назначение:** Обработка **новых записей** (`Store`), обновлений и удалений.
   - **Доступ:** Доступ к этой мапе **защищен `Mutex`** (`mu`), поскольку это обычная Go map, требующая синхронизации.
   - **Содержимое:** Хранит **все данные из `readonly`**, а также любые новые записи, которые еще не были "повышены" до `readonly`.

### Единый механизм обновления: Структура `entry`

Для обеспечения того, чтобы обе мапы видели одни и те же актуальные значения, используется косвенная адресация:

- **Хранение:** Обе `map`-ы (`readonly` и `dirty`) хранят не само значение, а **указатель** на структуру **`entry`**.

```go title:sync/map.go ln:87
// An entry is a slot in the map corresponding to a particular key.
type entry struct {
	p atomic.Pointer[any]
}
```

- **Обновление:** При обновлении значения достаточно атомарно обновить указатель **`p`** внутри структуры `entry`. Поскольку обе мапы ссылаются на один и тот же `entry`, они **автоматически** видят обновленное значение.

### Состояния элементов (`entry` States)

Состояние элемента определяется значением атомарного указателя **`p`** внутри структуры `entry`. Эти состояния позволяют `sync.Map` эффективно управлять элементами в двух внутренних `map`-ах (`readonly` и `dirty`), минимизируя блокировки. Существуют 3 состояния `entry`:

1. Нормальное состояние (Valid/Active)
   - **Условие:** **`p` указывает на реальное значение** (`p != nil` и `p != expunged`).
   - **Смысл:** Элемент является **валидным** и находится в активном использовании.
   - **Местоположение:** Элемент гарантированно существует в **`m.read.m[key]`** и, если `m.dirty` не `nil`, то и в **`m.dirty[key]`**.
   - **Обновление:** Значение элемента может быть **обновлено атомарно** (заменой указателя `p`), при условии, что `p != expunged`.
2. Состояние Удаления (Deleted)
   - **Условие:** **`p == nil`**.
   - **Смысл:** Элемент был **удален** (`Delete`), но физически не сразу исключен из мап.
   - **Местоположение:** Элемент может находиться как в `readonly`, так и в `dirty`. Если `m.dirty` не `nil`, то `m.dirty[key]` будет указывать на этот элемент.
   - **Механизм:** Удаление происходит путем **атомарной замены** указателя `p` на **`nil`**.
   - **Переход:** При следующем создании `m.dirty` (или продвижении) система **атомарно заменит `nil` на `expunged`**, и элемент будет удален из `m.dirty`.
3. Состояние Полного Исключения (Expunged)
   - **Условие:** **`p == expunged`** (специальное внутреннее значение-сентинел).
   - **Смысл:** Элемент **полностью удален из `m.dirty`**. Это конечное состояние удаленного элемента перед его полным физическим удалением.
   - **Местоположение:** Элемент **существует только в `m.read.m[key]`**, но **отсутствует** в `m.dirty`.
   - **Удаление:** Элементы в этом состоянии **полностью удаляются**, когда `m.dirty` замещает `m.read` (происходит "промоушн").
   - **Обновление/Возрождение:** Чтобы обновить элемент в состоянии `expunged` (или возродить его), необходимо сначала **вставить его обратно в `m.dirty[key]`**, и только потом атомарно обновить его значение.

#### Переходы между состояниями

- Элемент не может перейти из **Normal** ↔ **Expunged** напрямую; он **должен пройти через состояние Deleted**.
- **Возрождение (Revival):** Элементы в состояниях **Deleted** и **Expunged** могут быть **возвращены в Normal** (`p` устанавливается на новое значение), если тот же ключ снова добавляется (`Store`).

## Как работает `Load` в `sync.Map`

Операция `Load` (получение данных) всегда начинается с **быстрого пути** (fast path) — попытки чтения из `readonly` map без блокировки.

### 1. Быстрый путь (Fast Path)

`Load` сначала проверяет **`readonly`** map (через атомарный указатель `m.read`).

- Если ключ найден и элемент не помечен как удаленный (`p != nil` и `p != expunged`), значение **немедленно возвращается**.
- Это позволяет множеству горутин читать данные одновременно, **без использования мьютекса**.

```go title:sync/map.go ln:127
func (m *Map) Load(key any) (value any, ok bool) {
    read := m.loadReadOnly() // 1. Атомарно загружаем указатель на readOnly
    e, ok := read.m[key]     // 2. Ищем ключ в readOnly (быстро, без блокировки)
    if !ok && read.amended {
       // ... slow path ...
    }
    // ...
    return e.load() // 3. Если ключ найден: e.load() проверяет состояние p и возвращает значение
}
```

### 2. Медленный путь (Slow Path)

Если ключ **не найден** в `readonly` map, но установлен флаг **`read.amended`** (означает, что в `dirty` есть данные, которых нет в `readonly`), система переходит к медленному пути.

- **Блокировка:** Выполняется **захват мьютекса** (`m.mu.Lock()`).
- **Проверка `dirty`:** Идет поиск ключа в **`m.dirty` map**.
- **Учет промаха (Miss):** Независимо от того, найден ключ в `dirty` или нет, система **увеличивает счетчик `m.misses`** (`m.missLocked()`). Это сигнализирует о неактуальности `readonly` map.

```go title:sync/map.go ln:130
	// ...
    if !ok && read.amended {
       m.mu.Lock() // ЗАХВАТ МЬЮТЕКСА
       // ... повторная проверка readOnly (если m.dirty был повышен, пока ждали блокировку) ...
       if !ok && read.amended {
          e, ok = m.dirty[key] // Поиск в dirty map
          m.missLocked()       // Увеличение счетчика промахов
       }
       m.mu.Unlock() // ОСВОБОЖДЕНИЕ МЬЮТЕКСА
    }
    // ...
```

### 3. Продвижение `dirty` Map (Promotion)

- `sync.Map` использует счетчик **`m.misses`** для отслеживания промахов в `readonly` map.
- Если **количество промахов становится слишком большим**, это является триггером для **продвижения** (`promotion`):
  - `m.dirty` map **заменяет** собой `readonly` map.
  - Старая `readonly` map отбрасывается.
  - `m.dirty` map **сбрасывается** в `nil`.
- Этот механизм гарантирует, что `readonly` map остается относительно актуальной в сценариях с преобладанием чтения.

### 4. Проверка состояния элемента (`e.load()`)

Метод **`e.load()`** обеспечивает, что загружаемое значение не является удаленным:

```go title:sync/map.go ln:152
func (e *entry) load() (value any, ok bool) {
    p := e.p.Load()
    if p == nil || p == expunged { // Проверка, что элемент не в состоянии Deleted (nil) или Expunged
       return nil, false
    }
    return *p, true
}
```

### Главный вывод

Частое добавление новых ключей приводит к тому, что `readonly` map устаревает. Это вынуждает `Load` часто идти по **медленному пути** (с блокировкой и увеличением `m.misses`), что, в свою очередь, запускает дорогостоящую операцию **продвижения `dirty` map** для восстановления производительности.

## `Store`/`Swap`: Обновление и Возрождение

Операции `Store` и `Swap` следуют логике **быстрого/медленного пути** и могут либо добавить новый ключ, либо обновить/возродить существующий.

### 1. Быстрый путь (Fast Path)

- Если ключ находится в `readonly` map и **не находится в состоянии `Expunged`**, обновление происходит **атомарно** через `e.trySwap(&value)` **без блокировки**.

### 2. Медленный путь и Возрождение (`unexpungeLocked`)

Медленный путь (с мьютексом) запускается, если:

- Ключа нет в `readonly` map (нужно проверить `dirty`).
- Ключ есть в `readonly` map, но находится в состоянии **`Expunged`**.

**Возрождение (`Revival`):**

- Если элемент был **`Expunged`** (исключен из `dirty` map), требуется дополнительная работа. Метод **`e.unexpungeLocked()`** переводит его обратно в состояние **`Deleted` (`nil`)** и **вставляет** соответствующую `entry` обратно в **`m.dirty` map**.
- После этого `Swap` может безопасно обновить значение.

### Почему существует состояние `Expunged`?

Состояние `Expunged` (полное исключение) выполняет роль **промежуточного шага очистки** для "мягко удаленных" элементов.

1. **Предотвращение раздувания:** Если удаленные элементы (`p == nil`) останутся в обеих мапах, они будут **захламлять** `sync.Map`. `Expunged` позволяет **пакетно** очищать эти записи.
2. **Очистка в `dirty`:** После того как ключ становится `Expunged`, он **удаляется из `m.dirty`**. Это гарантирует, что при следующем **продвижении** (`promotion`) `dirty` map, все `Expunged` записи будут **полностью удалены** из новой `readonly` map.

### Переход из `nil` в `Expunged` (`dirtyLocked`)

Переход из состояния **`Deleted`** (`p == nil`) в состояние **`Expunged`** происходит при **инициализации `m.dirty`** (если она была `nil`).

```go title:sync/map.go ln:521
func (m *Map) dirtyLocked() {
    if m.dirty != nil { // 1. Проверяем, если dirty map уже инициализирована, то выходим
        return
    }

    read := m.loadReadOnly()
    m.dirty = make(map[any]*entry, len(read.m)) // 2. Создаем новую пустую dirty map

    for k, e := range read.m {
        // 3. Пытаемся перевести nil (Deleted) в expunged.
        if !e.tryExpungeLocked() { // Если tryExpungeLocked() НЕ удалось (т.е. p != nil),
            m.dirty[k] = e         // тогда копируем элемент в новую dirty map.
        }
    }
    // В итоге: все элементы с p==nil в readOnly map становятся expunged (p==expunged) и
    // не копируются в новую dirty map, которая теперь "чиста" от удаленных записей.
}
```

#### Разбор `Swap` (Важные шаги)

```go title:sync/map.go ln:358
func (m *Map) Swap(key, value any) (previous any, loaded bool) {
    read := m.loadReadOnly()
    if e, ok := read.m[key]; ok {
       if v, ok := e.trySwap(&value); ok { // БЫСТРЫЙ ПУТЬ: Атомарный swap, если не expunged
          // ... return ...
       }
    }

    m.mu.Lock() // МЕДЛЕННЫЙ ПУТЬ (с блокировкой)

    // ... повторная проверка read (если ключ был добавлен другой горутиной) ...

    if e, ok := read.m[key]; ok {
       if e.unexpungeLocked() { // Если элемент был Expunged, он переводится в nil
          m.dirty[key] = e      // и вставляется обратно в dirty map (Возрождение)
       }
       if v := e.swapLocked(&value); v != nil {
          // ... Обновление (для существующего ключа) ...
       }
    } else if e, ok := m.dirty[key]; ok {
       // ... Обновление (если ключ только в dirty map) ...
    } else {
       // Добавление НОВОГО ключа
       if !read.amended {
          m.dirtyLocked() // Создаем dirty map, если она nil, и маркируем readOnly как amended
          m.read.Store(&readOnly{m: read.m, amended: true})
       }
       m.dirty[key] = newEntry(value) // Запись нового ключа в dirty map
    }
    m.mu.Unlock()
    return previous, loaded
}
```

## `Delete`: Удаление данных

### 1. Удаление из `readonly` Map (Lock-Free)

- Удаление элемента из `readonly` map происходит **без блокировки** (`Mutex`).
- Используется **атомарный цикл Compare-And-Swap (CAS)**:
  - Указатель **`p`** элемента (`e.p`) атомарно заменяется на **`nil`** до тех пор, пока операция не завершится успешно (или пока элемент уже не будет удален другой горутиной).
- После этого элемент переходит в состояние **`Deleted`** (`p == nil`).

```go title:sync.map ln:328
func (e *entry) delete() (value any, ok bool) {
	for {
		p := e.p.Load()
		if p == nil || p == expunged { // Элемент уже удален
			return nil, false
		}
		if e.p.CompareAndSwap(p, nil) { // Атомарная замена на nil (Deleted state)
			return *p, true
		}
	}
}
```

### 2. Удаление из `dirty` Map (С блокировкой)

- Если ключ найден только в `dirty` map, элемент **сразу удаляется** из неё с помощью нативной функции `delete(m.dirty, key)`.
- Не требуется никаких специальных состояний (`nil`/`expunged`), что обеспечивает немедленную "чистую" утилизацию места.

## Механизм продвижения мапы (`Promotion`)

Продвижение `dirty` map в `readonly` map (когда `dirty` становится новой `readonly`) является **критическим механизмом очистки** и актуализации, который запускается двумя способами:

### 1. Через `Load()`

- **Триггер:** Продвижение запускается, когда счетчик **`m.misses`** (увеличивается при промахе в `readonly` и проверке `dirty` во время `Load`) достигает порогового значения.
- **Проблема:** Если код выполняет только **много `Store()` и `Delete()`**, но мало **`Load()`**, счетчик `misses` **не растет**, продвижение не происходит, и `dirty` map **раздувается** удаленными элементами (`p == nil`). Это может привести к утечке памяти (ключи не утилизируются сборщиком мусора).

### 2. Через `Range()`

- Операция `Range()` также может инициировать продвижение.
- Если `readonly` map устарела (`amended == true`), **система продвигает `dirty` map в `readonly`**, а затем **итерируется по новой, актуализированной `readonly` map**. Это гарантирует, что итерация происходит по максимально актуальному набору данных.

### Примечание о `Len()`

- `sync.Map` не имеет метода `Len()`. Это объясняется тем, что для получения **точного** количества элементов нужно было бы отфильтровывать удаленные элементы (`nil` и `expunged` указатели), которые все еще висят в мапе.
- Требуемый подсчет данных аналогичен итерации (`Range`), и для получения точного результата необходимо вручную использовать `Range()` и считать элементы.
